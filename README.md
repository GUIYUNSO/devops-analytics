# DevOps 分析插件

### 一句话问出代码仓库的风险、根因和趋势 —— AI 自动采集数据、推理分析、给出行动建议

一个 Codex / Claude Code 插件，让软件团队用自然语言完成工程质量分析：风险评分、bug 根因追溯、趋势检测、周报生成，全部在一个对话里完成。

## 它是什么

EIA 是一个 Codex 插件，由 6 个独立技能组成。你用自然语言提问，它自动完成以下流程：

1. **采集** — 从 git 历史和 GitHub 接口拉取提交、合并请求、议题、评审数据
2. **构建** — 将事件链接为生命周期时间线，构建实体关系图谱
3. **推理** — 风险评分、根因分析、趋势检测三种模式
4. **规划** — 将分析结果转化为按优先级排序的行动建议
5. **报告** — 用你的语言输出结论，附带具体数据证据

不需要写代码，不需要配看板，直接问就行。

## 快速安装

```bash
git clone https://github.com/GUIYUNSO/devops-analytics.git
cp -r devops-analytics/plugins/devops-analytics ~/.codex/plugins/
rm -rf devops-analytics
```

然后在 `~/.agents/plugins/marketplace.json` 的 `plugins` 数组末尾加：

```json
{
  "name": "devops-analytics",

  "policy": { "installation": "AVAILABLE", "authentication": "ON_INSTALL" },

```

## 6 个技能详解

### eia — 编排器

**职责**：接收用户问题，判断意图，选择分析路径，编排后续技能执行。

**具体能力**：
- 自动识别用户角色（开发者 / 技术负责人 / 产品经理 / 测试），根据角色调整输出详略
- 自动检测语言，中文提问中文回答，英文提问英文回答
- 根据问题关键词分类为 6 种意图之一，每种意图走不同流程
- 读取历史记忆（`~/.eia/memory.md`），跨会话积累分析上下文

**意图路由规则**：

| 意图 | 触发词 | 执行的流程 |
|------|--------|-----------|
| 调试 | 出bug了、error、crash、失败 | 采集 → 构建 → 根因分析 → 报告 |
| 调查 | 为什么、why、分析、根因、trace | 采集 → 构建 → 全部推理 → 规划 → 报告 |
| 监控 | 状态、status、health、看看 | 采集 → 构建 → 风险评分 → 报告 |
| 预测 | 预测、risk、风险、会不会 | 采集 → 构建 → 风险+趋势 → 规划 → 报告 |
| 报告 | 报告、周报、月报、汇总 | 采集 → 构建 → 趋势分析 → 规划 → 报告 |
| 优化 | 优化、改进、怎么办、improve | 采集 → 构建 → 全部推理 → 规划 → 报告 |

---

### eia-collect — 数据采集

**职责**：从 git 和 GitHub 接口拉取原始数据，统一为标准事件格式。

**具体能力**：
- **git log**：自动采集提交历史（作者、时间、消息、文件变更、增删行数）
- **GitHub 接口**：拉取合并请求（标题、作者、评审人、状态、大小）、议题、评审评论
- **事件归一化**：不同来源的数据统一为 `{type, actor, repo, timestamp, payload}` 格式
- **模块识别**：按文件路径前缀自动归类模块（如 `src/payment/` → 支付模块）
- **变更频率计算**：统计每个文件在时间窗口内的变更次数
- **归属识别**：按提交数量识别每个模块的主要贡献者

**输出示例**：
```json
{
  "events": [...],
  "modules": {
    "src/payment": { "commits": 12, "contributors": ["alice"], "churn": 0.85, "bug_rate": 3.2 }
  }
}
```

---

### eia-build — 知识构建

**职责**：将采集到的事件构建为两种结构化知识。

**时间线构建**：
- 通过提交消息中的议题编号、合并请求中的提交哈希将事件链接为生命周期链
- 计算每个链的交付周期（议题创建到部署）、开发周期（首次提交到部署）、评审时间
- 输出瓶颈阶段（哪个环节耗时最长）

**知识图谱构建**：
- 实体：开发者、模块、合并请求、提交、议题、Bug、部署
- 关系：编写、提交、评审、归属、修改、解决、引入、导致
- 支持查询：给定实体找所有关联实体、两个实体之间的路径、模块的Bug数量

### eia-reason — 统一推理

**职责**：根据意图选择分析模式，输出结构化发现。

#### 模式 1：风险评分

为每个合并请求和模块打分（0-1），评分因子：

| 因子 | 权重 | 评分规则 |
|------|------|---------|
| 合并请求大小（代码行数） | 高 | >400 行=0.7，>800 行=0.9 |
| 代码变更频率 | 高 | 直接取变更频率分值 |
| 评审深度 | 中 | <10 分钟=0.8，<1 小时=0.5 |
| 评审人数量 | 中 | 仅 1 人=0.7 |
| 历史Bug | 高 | 模块Bug率 >2/100 次提交=0.8 |
| 单点依赖 | 中 | 仅 1 人维护=0.6 |

**上下文修正**：机器人提交（dependabot）风险减半；纯文档合并请求置为 0.1。

输出风险等级：低（0-0.3）/ 中（0.3-0.6）/ 高（0.6-0.8）/ 严重（0.8-1.0）

#### 模式 2：根因分析

对每个Bug执行反向追溯：

```
Bug → 哪次部署引入的？→ 哪个合并请求在那次部署里？→ 哪个提交改了相关代码？→ 根因分类
```

根因分类：代码Bug / 配置变更 / 依赖问题 / 基础设施 / 流程缺失

**链路断裂处理**：每一步都有降级方案，链路不完整时标记 `数据缺口`，不猜测，降级输出已知部分。

#### 模式 3：趋势分析

对比当前周期与上一周期：

| 指标 | 计算方式 | 显著性阈值 |
|------|---------|-----------|
| Bug数量 | 本期对比上期 | >50% 变化=严重 |
| 交付周期 | 中位数对比 | >50% 变化=严重 |
| 合并请求大小 | 中位数对比 | >50% 变化=严重 |
| 部署频率 | 每周部署次数 | — |
| 评审时间 | 从打开到批准的中位数 | — |

**评审文化分析**：对比待处理合并请求和已合并请求的评审覆盖率，区分"临时积压"和"长期无评审文化"。

### eia-planner — 行动规划

**职责**：将推理结果转化为按优先级排序的行动建议。

**优先级定义**：
- **[P0]** — 阻塞发布或安全风险，本周内处理
- **[P1]** — 降低质量或效率，本迭代处理
- **[P2]** — 改进机会，放入待办

**行动规则库**（部分）：

| 发现 | 建议 |
|------|------|
| 合并请求 > 500 行 | 拆分为更小的合并请求 |
| 单人评审高风险合并请求 | 增加评审人 |
| 评审时间 < 5 分钟 | 重新评审 |
| 模块变更频率 > 10 次/周 | 计划重构 |
| 单点依赖 = 1 | 知识转移 |
| 根因为代码Bug | 添加回归测试 |
| Bug数量严重上升 | 检查合并请求大小，增加测试覆盖 |

**角色适配**：开发者看到文件级建议，产品经理看到时间线建议，技术负责人看到团队级建议。

---

### eia-report — 报告生成

**职责**：将所有上游结果组织为用户可读的输出。

**两种输出格式**：

**对话模式**（默认）：
```
[一句话结论]

[关键数据 — 3-5 个，带数字]

[P0/P1/P2 行动项]

[一个跟进问题]
```

**报告模式**（报告意图时）：
```markdown
# 工程效率报告 — {时间段}

## 概况
一段话总结

## 关键指标
| 指标 | 本期 | 上期 | 变化 | 状态 |

## 发现
1. **标题** — 说明 [证据: 合并请求 #xxx]

## 行动建议
[P0] 建议内容
[P1] 建议内容
[P2] 建议内容
```

**输出规则**：
- 系统性表述，不针对个人（「项目存在单点活跃风险」而非「某人有 50% 合并请求」）
- 每个数字带上下文（「86% 未评审」而非「很多合并请求没评审」）
- 每个发现配行动建议
- 末尾追加流程追踪和记忆写入

## 使用示例

### 调试问题
```
你：为什么支付模块最近总是超时？
AI：自动执行 采集 → 构建 → 根因分析 → 报告
输出：定位到合并请求 #89 的提交 abc123 改了 validator.go:142，缺少接口超时的空值检查
```

### 风险评估
```
你：这个模块风险怎么样？
AI：自动执行 采集 → 构建 → 风险评分 + 趋势分析 → 规划 → 报告
输出：支付模块风险分=0.78（高），主要因子：变更频率=0.85、Bug率=3.2、单点依赖=1
```

### 周报生成
```
你：给这周的工程效率做个报告
AI：自动执行 采集 → 构建 → 趋势分析 → 规划 → 报告
输出：Markdown 格式的完整工程效率报告，含指标对比、发现、行动建议
```

### 根因追溯
```
你：为什么最近线上Bug越来越多？
AI：自动执行 采集 → 构建 → 全部推理 → 规划 → 报告
输出：Bug +50% 环比，主要集中在支付模块，根因是 3 个大合并请求缺少评审
```

## 工作原理

```
用户提问
    │
    ▼
┌─ eia ────────────────┐
│  推断角色、意图、语言   │
└──────────┬───────────┘
           ▼
    采集 → 构建 → 推理 → 规划 → 报告
    │       │      │       │      │
    数据    时间线  风险    行动项  输出
    拉取    图谱    根因    优先级  对话
            趋势
```

6 个技能独立运行，通过 `previous_output` 传递数据，每个有严格输入输出契约。

## 常见问题

**需要联网吗？**

Git 数据完全本地采集。GitHub 接口需要 `GITHUB_TOKEN`，仅用于拉取合并请求/议题/评审数据，不做其他网络请求。

**没有 GITHUB_TOKEN 能用吗？**

能。Git 数据开箱即用。没有令牌时跳过 GitHub 接口，分析范围限于本地 git 历史。核心功能（风险评分、根因分析、趋势检测）仍然可用。

**记忆存在哪？安全吗？**

`~/.eia/memory.md`，纯文本仅追加文件。只记录分析摘要（如「支付模块 30 天 3 个Bug」），不记录代码内容。你可以随时查看、编辑或删除。

**支持哪些意图？**

6 种：调试、调查、监控、预测、报告、优化。AI 根据你的问题自动选择，也可以在提问中明确指定。

**能分析多仓库吗？**

目前按单仓库设计。多仓库场景可以通过指定不同仓库多次运行。

**输出可以导出吗？**

报告直接以 Markdown 格式输出在对话中，可以复制粘贴到任何地方。也可以让 AI 写入文件：「把报告写到 report.md」。
